# SQLite VSCode 插件 — 实施方案

> 配套《开发计划.md》。本文聚焦技术架构、关键设计与工程落地细节。

## 1. 整体架构

VSCode 扩展由两个相互隔离的运行环境组成，通过 `postMessage` 通信：

```
┌─────────────────────────────────────────────────────────────┐
│  扩展宿主 (Extension Host, Node.js)                            │
│  - extension.ts: activate / 注册 CustomEditorProvider         │
│  - SqliteEditorProvider: 文件读写、脏标记、保存、备份、revert  │
│  - 真实文件 IO：vscode.workspace.fs (替代 uTools preload fs)   │
└───────────────▲───────────────────────────┬──────────────────┘
                │  postMessage (异步)         │
                │  ── 协议消息 ──             ▼
┌───────────────┴───────────────────────────────────────────────┐
│  Webview (Chromium 沙箱)                                       │
│  - React 19 + Ant Design 6 UI（移植自参考项目）               │
│  - sql.js (WASM)：在内存加载/操作数据库字节                    │
│  - Monaco：SQL 编辑器                                          │
│  - bridge.ts: 实现"原 customApis"语义，底层走 postMessage      │
└───────────────────────────────────────────────────────────────┘
```

**职责划分**
- **宿主**：唯一能碰磁盘的一方。负责把文件字节交给 Webview、接收 Webview 的"内容已变更/请保存"请求、把字节写回磁盘，并参与 VSCode 的脏标记/撤销/备份生命周期。
- **Webview**：所有业务逻辑与 UI。用 sql.js 在内存里加载宿主给的字节，执行查询/变更；变更后导出新字节回传宿主。

## 2. CustomEditorProvider 设计

采用 `vscode.window.registerCustomEditorProvider` + **`CustomEditorProvider`（带自定义文档模型）**，而非只读的 `CustomReadonlyEditorProvider`，以支持编辑与保存。

### 2.1 贡献点（package.json）

```jsonc
"contributes": {
  "customEditors": [
    {
      "viewType": "sqliteManager.editor",
      "displayName": "SQLite 管理器",
      "selector": [
        { "filenamePattern": "*.db" },
        { "filenamePattern": "*.sqlite" },
        { "filenamePattern": "*.sqlite3" }
      ],
      "priority": "default"
    }
  ]
}
```

### 2.2 自定义文档 `SqliteDocument`

实现 `vscode.CustomDocument`，内部维护：
- `uri`：文件路径。
- `documentData: Uint8Array`：当前（可能已修改的）数据库字节。
- `savedData`：上次保存时的字节（用于 revert/对比）。

需要实现的 Provider 接口方法：

| 方法 | 作用 |
|------|------|
| `openCustomDocument(uri, openContext)` | 读取磁盘字节（或 `backupId` 备份），构造 `SqliteDocument` |
| `resolveCustomEditor(document, panel)` | 配置 Webview（HTML、CSP、资源根），建立消息通道，把初始字节发给 Webview |
| `saveCustomDocument(document)` | 把当前字节写回原文件 |
| `saveCustomDocumentAs(document, dest)` | 另存为 |
| `revertCustomDocument(document)` | 重新从磁盘读取，刷新 Webview |
| `backupCustomDocument(document, ctx)` | 写备份文件，支持热退出 |
| `onDidChangeCustomDocument` (Event) | 触发脏标记；携带 undo/redo 回调 |

### 2.3 脏标记与保存流程

1. Webview 内用户改了数据（增/删/改/建表）→ sql.js 导出新字节 → `postMessage({ type: 'contentChanged', data })`。
2. 宿主收到后更新 `document.documentData`，并 `fire` `onDidChangeCustomDocument`（携带 label + undo/redo）→ VSCode 显示编辑器为"脏"。
3. 用户 `Ctrl/Cmd+S` → VSCode 调 `saveCustomDocument` → 宿主 `workspace.fs.writeFile(uri, documentData)`。
4. `revert` → 宿主读磁盘字节 → 推送 Webview 重新用 sql.js 加载。
5. `backup` → 写入 `ctx.destination`，供热退出恢复。

> 说明：sql.js 的 undo/redo 难以做到字节级精确，本期 undo 回调采用"快照回退"策略（保存变更前后的字节快照），保证脏标记与保存正确即可，编辑历史精度后续优化。

## 3. postMessage 桥接协议（替代 customApis）

参考项目里所有平台能力都走 `window.customApis`（同步）。新方案在 Webview 提供一个 **`bridge`** 模块，对外暴露与原 `customApis` 语义对齐的方法，但内部是 **异步 + postMessage**。

### 3.1 消息信封

```ts
// Webview → Host：请求
interface RequestMsg { kind: 'req'; id: string; method: string; params: any[] }
// Host → Webview：响应
interface ResponseMsg { kind: 'res'; id: string; ok: boolean; result?: any; error?: string }
// Host → Webview：主动推送（初始数据、revert 后刷新等）
interface PushMsg { kind: 'push'; type: 'init' | 'reload'; data: Uint8Array }
// Webview → Host：内容变更通知（标脏）
interface ChangedMsg { kind: 'changed'; data: Uint8Array; label?: string }
```

`bridge` 内部维护 `Map<id, {resolve, reject}>`，发请求时生成 id、登记 Promise，收到同 id 的 `res` 后兑现。

### 3.2 桥接方法（Webview 侧 API）

对齐原 customApis 中本期需要的能力：

| 方法 | 走向 | 说明 |
|------|------|------|
| `getInitialData()` | Host 主动 push `init` | 打开文件时宿主直接推首批字节，Webview 缓存 |
| `notifyChanged(data)` | Webview → Host `changed` | 通知内容变更，触发脏标记 |
| `getFileInfo()` | req → res | 文件大小、修改时间（宿主 `fs.stat`） |
| `openFileLocation()` | req → res | `vscode.commands` 在资源管理器中定位文件 |
| `getSettings()/saveSettings()` | req → res | 走 `globalState`（替代 uTools dbStorage） |

> 注意：原 `readFile/writeFile/selectFile/createSqliteDatabase` 等针对"多文件管理"的方法本期不再需要——单文件由宿主直接 push 字节、由 `notifyChanged` 回写。`SQLiteHelper` 相应改造（见 §5）。

### 3.3 二进制传输

字节以 `Uint8Array` 直接随结构化克隆传输（VSCode webview `postMessage` 支持 transferable / 结构化克隆）。若遇兼容问题，退回 base64 字符串。

## 4. 资源处理：sql.js 与 Monaco

### 4.1 sql.js WASM

- 将 `sql-wasm.wasm` 放入 Webview 资源目录（如 `media/`）。
- 宿主在生成 Webview HTML 时，用 `panel.webview.asWebviewUri(wasmUri)` 得到可访问 URI，注入到页面（`window.__SQL_WASM_URI__`）。
- `SQLiteHelper` 的 `initSqlJs({ locateFile: () => window.__SQL_WASM_URI__ })` 读取该 URI。

### 4.2 Monaco

- 采用 `@monaco-editor/react` + `monaco-editor`。
- Monaco 的 worker 与静态资源随 Vite 打包进 Webview 产物；在 Webview 中通过注入的 base URI 定位（`monaco-loader` 适配）。
- 配置 `MonacoEnvironment.getWorkerUrl` 指向打包后的 worker 资源（经 `asWebviewUri`）。

### 4.3 CSP（内容安全策略）

Webview HTML `<meta http-equiv="Content-Security-Policy">`：
- `default-src 'none'`
- `img-src ${webview.cspSource} data:`
- `style-src ${webview.cspSource} 'unsafe-inline'`（Ant Design CSS-in-JS 需要）
- `script-src 'nonce-${nonce}' 'wasm-unsafe-eval'`（sql.js wasm 需要 wasm-eval）
- `font-src ${webview.cspSource}`
- `connect-src ${webview.cspSource}`
- 所有脚本标签带 `nonce`。

## 5. SQLiteHelper 改造

参考项目 `SQLiteHelper` 通过同步 `customApis.readFile/createSqliteDatabase` 读写文件。改造点：

- **去掉文件路径读取**：构造时不再传 path 读盘，而是接收**字节**（由宿主 push 的 init 数据）。
- `init(bytes)`：直接用传入字节 `new SQL.Database(bytes)`。
- `save()`：改为 `bridge.notifyChanged(this.export())`，不再直接落盘。
- `reload(bytes)`：用宿主 revert/reload 推送的新字节重建实例。
- 其余查询/分页/schema/建表 SQL/INSERT 生成等**纯内存逻辑全部原样保留**。

## 7. 构建与脚本

```jsonc
"scripts": {
  "build:webview": "vite build --config webview/vite.config.ts",   // → media/
  "build:ext": "node esbuild.js",                                   // → dist/extension.js
  "build": "npm run build:webview && npm run build:ext",
  "watch:webview": "vite build --watch --config webview/vite.config.ts",
  "watch:ext": "node esbuild.js --watch",
  "package": "npm run build && vsce package"
}
```

- Webview Vite 产物输出到 `media/`（`base: './'`，相对路径，便于 `asWebviewUri` 重写）。
- 宿主 esbuild：`platform: node`、`external: ['vscode']`、`format: cjs`。

## 8. 调试方式

1. `npm run build`（或并行 `watch:webview` + `watch:ext`）。
2. VSCode 中按 `F5` 启动 **Extension Development Host**。
3. 在新窗口里打开一个含 `.db`/`.sqlite`/`.sqlite3` 文件的目录，点击该文件。
4. 应打开"SQLite 管理器"自定义编辑器，看到表列表与数据。
5. 改数据 → 编辑器标题出现脏标记圆点 → `Ctrl/Cmd+S` 保存 → 重开验证。
6. Webview 内部 UI 调试：命令面板 `Developer: Open Webview Developer Tools`。
